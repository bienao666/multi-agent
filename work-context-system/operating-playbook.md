# Operating Playbook

这份手册定义 `.work-context/` 的日常运行方式。目标是让上下文持续更新，而不是一次性建完就荒废。

## 日常维护节奏

| 触发场景 | 应更新内容 |
| --- | --- |
| 开始一个新项目 | 新建 `projects/<project>.md`，并更新 `index.md` |
| 参加重要会议 | 新建 `events/YYYY-MM-DD-<topic>.md` |
| 收到关键反馈 | 更新相关 `people/<role>.md` 和事件文件 |
| 项目目标或排期变化 | 更新项目库和最近事件 |
| 形成稳定产出格式 | 更新或新增 `templates/<template>.md` |
| 完成重要汇报或方案 | 可选保存到 `outputs/`，并沉淀模板经验 |

## 推荐读取顺序

让 AI 生成产出时，按这个顺序读取：

```text
index.md
  -> 相关人物画像
  -> 相关项目文件
  -> 相关事件文件
  -> 产出模板
  -> 历史输出
```

不要默认读取整个 `.work-context/`，避免上下文过大和无关信息干扰。

## 标准版到人物版

重要材料建议分两步生成：

1. **标准版**
   - 基于事实、项目和事件生成。
   - 不刻意迎合某个读者。
   - 用于保证信息完整和逻辑正确。

2. **人物版**
   - 基于人物画像改写。
   - 调整重点、结构、语气和风险表达。
   - 增加对方可能关心的问题和建议回答。

推荐提示词：

```text
请先基于项目库和事件库生成标准版报告。
然后读取指定人物画像，把标准版改写成面向该人物的版本。
最后说明改写时调整了哪些重点，以及为什么。
```

## 质量检查清单

生成汇报、方案或复盘前，检查：

- 是否明确读者是谁。
- 是否读取了对应人物画像。
- 是否读取了项目背景和当前进展。
- 是否读取了最近关键事件。
- 是否使用了正确产出模板。
- 是否先生成标准版。
- 是否按读者关注点改写。
- 是否避开人物画像中的忌讳点。
- 是否模拟了可能追问。
- 是否给出下一步动作。

## 追问模拟流程

每次生成重要材料后，都让 AI 模拟追问：

```text
请基于读者画像，模拟对方可能提出的 5 到 10 个问题。
每个问题给出：
1. 对方为什么会问。
2. 建议回答。
3. 是否需要提前补充到正文里。
```

## 更新人物画像

收到反馈后，用这个提示词更新人物库：

```text
请根据下面的新反馈，更新对应人物画像。
只更新能被材料支持的内容，不做主观猜测。
请输出：
1. 新增关注点
2. 新增表达偏好
3. 新增忌讳点
4. 常问问题变化
5. 需要写入人物画像的最终文本
```

## 更新项目库

项目发生变化后，用这个提示词：

```text
请根据下面材料更新项目库。
重点更新：
1. 当前进展
2. 目标变化
3. 风险变化
4. 相关人物
5. 当前待办
```

## 更新事件库

会议或沟通后，用这个提示词：

```text
请把下面材料整理成事件记录。
要求：
1. 只记录事实和明确结论。
2. 分开记录分歧、风险和后续动作。
3. 标注相关人物态度。
4. 给出需要同步更新的人物库和项目库条目。
```
